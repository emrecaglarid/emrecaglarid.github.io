Lore

Personal project · 2025–present · MCP · Cloudflare Workers · Prototype infrastructure

When AI coding tools like Claude Code made it fast to build working prototypes, a new friction appeared on the other end: collecting feedback on them. You'd ship to Vercel, drop the link in Slack, and get responses scattered across threads, voice notes, and screenshots. Nobody had the full picture. Nobody could easily act on it.

I built Lore to close that loop.

Lore is an MCP server that adds a structured feedback layer to any deployed prototype. When I push a prototype through Lore, Claude annotates each meaningful element with a semantic name (hero-section, skip-button, auth-form). Comments from reviewers attach to those names. When the prototype gets redesigned, comments follow the element, not the layout. Nothing gets lost between iterations.

The second thing Lore does is write a semantic map alongside the prototype, a record of what each element is for, captured at the moment the design decision is being made. When a reviewer opens the prototype to comment, they see the intent behind what they're looking at. When I come back to Claude later, it can read the full history of why the prototype is the way it is and apply feedback with that context intact. Design decisions that normally live in Slack threads or in someone's head get written down automatically.

Reviewers need no account and nothing installed. They open the URL, click into comment mode, and leave feedback on the live page. The link is the tool.

Workflow

1. Build

Tell Claude to build a prototype or push an existing one. Lore automatically captures the semantic map, adds markup, and registers it.

2. Share

Get a tokenised review link per environment (local dev server, staging, production) that reviewers can open without logging in.

3. Review

Fetch all comments grouped by screen, element, or flow. Comments from any environment are unified in one feed.

Installation

One command installs Lore across all projects: 

claude mcp add lore -s user -- npx -y @emrecaglarid/lore-mcp

Restart Claude Code, and the Lore server appears in the MCP list. It works on any deployment (Vercel, Netlify, GitHub Pages, localhost) with no opinions about how you build or where you host.

Adoption

Two teams at Supermetrics are already using Lore in production: the Data Activation team (for the stream creation and connector prototypes) and the Storage team (for Storage 2.0 stream editor). The dashboard shows real prototypes with active feedback loops across multiple environments.

An unexpected benefit: Lore also helped designers who were new to AI-assisted prototyping by collecting all their deployed prototypes, whether running on localhost or deployed to staging and production, in a single dashboard. Instead of losing track of prototype URLs across Vercel projects, local servers, and Slack threads, everything is registered and accessible from one place.

I also built a macOS menu bar companion app that gives quick access to all registered prototypes: open in Claude Code, start a local server, copy the share link, or check new comments, all without leaving whatever you're working on.

Lore is published as an MCP package on npm. The API runs on Cloudflare Workers. The goal was infrastructure that disappears into the workflow, not another platform to manage.

Designer Mode for Agents

Internal tool · 2025–present · React · LLM integration · Agentic UX

At Supermetrics, we discuss agentic UX constantly: when should the agent act autonomously, when should it ask, how cautious should it be, how verbose. But after those discussions, most designers can't experiment with tangible artifacts. Designers are used to building and testing, sketching, prototyping, iterating with real users. For agent behaviour, there's no equivalent. The gap between "we should make the agent more cautious here" and actually experiencing what that feels like in the product is too wide.

I built Designer Mode to close that gap. It's a sidebar panel embedded directly in the product's frontend that exposes agent behaviour as a designable surface, the same way a component library exposes visual properties.

Designer Mode: recording of the agent sidebar embedded in the field mapper, running against the staging backend.

How it works

Designers create custom agent variants, each with its own system prompt that defines the agent's personality, caution level, verbosity, and action patterns. The prompt is editable directly in the sidebar, or the tool can automatically adjust it based on the designer's feedback, iterating toward the right behaviour without requiring prompt engineering expertise.

The Compare tab runs the same user input against multiple variants simultaneously, showing how each one responds side by side. A "fast" variant that acts without asking, a "careful" variant that confirms every step, a "technical" variant that uses domain jargon. Designers can see the difference immediately and decide which behaviour fits the context.

Everything runs against the actual staging backend through a proxy. The agent reads real data, proposes real actions, and surfaces real consequences. When a variant is validated, the prompt exports as a markdown file, ready to hand off to development. The prompt becomes a design deliverable, not just a prototype artifact.

Why it matters

Designers can now craft agentic UX the same way they craft any other interaction: by building variants, comparing alternatives, and testing with real users. They can share specific agent variants with consultants or customers to gather insights on which behaviour feels right for different use cases.

The tool also helps teams converge. After a discussion about how an agent should handle a risky action, a designer can build two variants in ten minutes and test them against real data. A concrete comparison replaces opinion. That changes the quality of the conversation.
